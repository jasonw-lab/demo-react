# ADR-001: UI技術スタックの選定（Next.js + shadcn/ui）

## ステータス

✅ **採用** (Accepted)

## 日付

2026-01-24

## コンテキスト

ホテル予約デモアプリケーションのフロントエンド技術を選定する必要がある。

### 要求事項

1. **即座にプロフェッショナル感が伝わる洗練されたUI**
2. **モダンな技術スタックによる開発効率**
3. **デモとして10秒以内に価値が伝わること**
4. **レスポンシブ対応とアクセシビリティ**
5. **将来的な拡張性とメンテナンス性**

### 制約条件

* バックエンドAPIは存在しない（フロントエンドのみのデモ）
* 短期間で高品質なUIを実現する必要がある
* 型安全性と保守性が重要

## 決定事項

以下の技術スタックを採用する：

### コアフレームワーク

* **Next.js 14+ (App Router)** - フロントエンドフレームワーク
* **TypeScript** - 開発言語
* **React 18+** - UIライブラリ（Next.jsに含まれる）

### UIコンポーネントライブラリ

* **shadcn/ui** - UIコンポーネントシステム
* **Tailwind CSS** - スタイリング
* **Radix UI** - プリミティブコンポーネント（shadcn/uiの基盤）
* **Lucide React** - アイコンライブラリ

### 補助ライブラリ

* **React Hook Form** - フォーム管理
* **Zod** - スキーマバリデーション
* **date-fns** - 日付操作
* **clsx / tailwind-merge** - クラス名管理

### 状態管理（Optional）

* **Zustand** - グローバル状態管理（必要に応じて）

## 理由

### Next.js 14+ (App Router) を選んだ理由

#### 利点

1. **最適化された開発体験**
   - ファイルベースルーティング
   - 組み込みの画像最適化（`next/image`）
   - 自動コード分割

2. **パフォーマンス**
   - Server Components によるパフォーマンス最適化
   - Streaming SSR による高速初期表示
   - 自動的な静的最適化

3. **SEO対応**
   - サーバーサイドレンダリング（SSR）
   - 静的サイト生成（SSG）
   - メタデータ管理

4. **デプロイの容易性**
   - Vercel への即座のデプロイ
   - プレビュー環境の自動生成

#### デメリット

* 学習曲線がやや高い（App Routerは比較的新しい）
* ただし、ドキュメントが充実しており、デモレベルでは問題なし

### shadcn/ui を選んだ理由

#### 利点

1. **プロフェッショナルなデザイン**
   - 洗練されたUIコンポーネント
   - 一貫したデザインシステム
   - モダンなビジュアルスタイル

2. **カスタマイズ性**
   - コンポーネントのソースコードをプロジェクトにコピー
   - 完全なカスタマイズが可能
   - ライブラリへの依存が少ない

3. **アクセシビリティ**
   - Radix UI ベースで ARIA 対応
   - キーボードナビゲーション対応
   - WCAG 2.1 AA準拠

4. **Tailwind CSS との統合**
   - Tailwind CSS によるスタイリング
   - デザイントークンの統一
   - レスポンシブ対応が容易

5. **開発効率**
   - コピー&ペーストで即座に使用可能
   - TypeScript完全対応
   - 豊富なコンポーネント（50+）

#### デメリット

* 各コンポーネントをプロジェクトにコピーする必要がある
* ただし、必要なコンポーネントのみ追加できるため、バンドルサイズは最小限

### TypeScript を選んだ理由

#### 利点

1. **型安全性**
   - コンパイル時のエラー検出
   - モックデータの型定義
   - IntelliSenseによる開発効率向上

2. **保守性**
   - リファクタリングが安全
   - ドキュメントとしての役割
   - 大規模化時のスケーラビリティ

3. **shadcn/ui との親和性**
   - すべてのコンポーネントがTypeScript対応
   - 型推論による開発体験の向上

## 検討した代替案

### 代替案1: Material-UI (MUI)

* **メリット**: 成熟したコンポーネントライブラリ、豊富なドキュメント
* **デメリット**: やや古めのデザイン、カスタマイズが複雑、バンドルサイズが大きい
* **却下理由**: 「即プロ感」という要件に対してデザインが古めに感じられる

### 代替案2: Ant Design

* **メリット**: 企業向けUI、豊富なコンポーネント
* **デメリット**: 中国発のデザイン言語、日本語対応が不完全
* **却下理由**: デザインがビジネス寄りでBooking系のコンシューマー向けUIに不適

### 代替案3: Chakra UI

* **メリット**: アクセシビリティ重視、優れたDX
* **デメリット**: カスタマイズにはテーマ設定が必要、やや重い
* **却下理由**: shadcn/uiの方がモダンでカスタマイズしやすい

### 代替案4: Vite + React

* **メリット**: 高速なビルド、シンプルな設定
* **デメリット**: ルーティング、画像最適化など自前で実装が必要
* **却下理由**: Next.jsの方が総合的な機能が充実

## 影響

### ポジティブ

1. **開発速度の向上**
   - shadcn/uiのコンポーネントで実装時間を大幅削減
   - Next.jsの機能で開発体験が向上

2. **UI品質の保証**
   - プロフェッショナルなデザインが最初から利用可能
   - アクセシビリティが標準で担保される

3. **パフォーマンス**
   - Next.jsの最適化機能により高速な表示
   - 画像最適化による帯域削減

4. **将来の拡張性**
   - バックエンドAPI追加時にスムーズに移行可能
   - TypeScriptによる型安全な開発

### ネガティブ

1. **学習コスト**
   - Next.js App Routerは比較的新しい
   - shadcn/uiのコンポーネントカスタマイズ方法の理解が必要

2. **初期セットアップ**
   - shadcn/ui CLIでのコンポーネント追加
   - Tailwind CSSの設定

### 技術的負債の回避

* コンポーネントライブラリへの強い依存を避ける（shadcn/uiはコピー方式）
* TypeScriptにより将来的なリファクタリングリスクを低減
* Next.jsの標準的な機能を使用し、独自実装を最小化

## 実装ガイドライン

### プロジェクト初期化

```bash
npx create-next-app@latest booking-hotel --typescript --tailwind --app
cd booking-hotel
```

### shadcn/ui セットアップ

```bash
npx shadcn-ui@latest init
```

### 推奨コンポーネント

デモで使用する主要コンポーネント：

* `Card`, `Button`, `Input`, `Form`
* `Calendar`, `Select`, `Slider`
* `Dialog`, `Sheet`, `Tabs`
* `Carousel`, `Badge`, `Avatar`
* `Skeleton`, `Separator`

### フォルダ構成

```
src/
├── app/               # Next.js App Router
│   ├── page.tsx      # トップページ
│   ├── search/       # 検索結果
│   └── hotels/       # ホテル詳細
├── components/
│   ├── ui/           # shadcn/ui コンポーネント
│   └── features/     # 機能別コンポーネント
├── lib/
│   ├── types/        # TypeScript型定義
│   └── data/         # モックデータ
└── styles/           # グローバルスタイル
```
